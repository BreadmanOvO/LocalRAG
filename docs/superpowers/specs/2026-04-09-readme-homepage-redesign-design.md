# README 首页重构设计

## 1. 背景
当前仓库只有 `README.md`，内容是一份覆盖项目定位、版本规划、技术决策、风险、简历表达、学习资料的综合规划文档。信息完整，但首页承担了过多职责，导致“当前在做什么、下一步做什么、去哪里继续看细节”不够清晰。

本次重构的目标不是削弱内容，而是重新分层：把 `README.md` 变成面向后续开发的首页导航，把详细规划下沉到 `docs/`。

## 2. 设计目标
1. 让首页优先服务当前开发推进，而不是一次性承载全部背景材料。
2. 同时展示“当前现状 + 近期计划”，并明确区分 `已完成 / 进行中 / 计划中`。
3. 让用户能一眼看清当前阶段、下一步任务以及去哪里查看详细方案。
4. 保留项目全貌，避免因首页过度压缩导致后续开发跑偏。

## 3. 非目标
1. 不在本次设计中补写不存在的运行命令、脚本或能力说明。
2. 不把规划目标改写成既成事实。
3. 不在首页保留大段简历表达、面试亮点和长参考资料列表。
4. 不改变项目技术路线本身，只调整文档信息架构。

## 4. 受众与使用场景
### 主要受众
仓库作者本人，用于后续按版本推进项目开发。

### 次要受众
未来可能查看仓库的面试官、协作者或自己回看时的阅读者。

### 核心使用场景
1. 打开首页后快速确认项目定位。
2. 立即知道当前处于哪个版本阶段。
3. 明确最近应该优先做哪些任务。
4. 需要细节时，快速跳转到对应文档继续阅读。

## 5. 信息架构设计
### 5.1 README 首页定位
`README.md` 改造成“开发驾驶舱型首页”，承担以下职责：
- 简洁说明项目目标与范围。
- 呈现当前状态总览。
- 明确当前阶段与下一步。
- 提供精简路线图。
- 提供详细文档导航。

### 5.2 详细文档下沉策略
首页不再承载所有展开内容，以下信息迁移到 `docs/`：
- 详细版本规划与任务拆解。
- 评估框架与指标定义。
- 简历表达与面试亮点。
- 参考资料与学习路径。

## 6. README 新结构
建议新的 `README.md` 使用以下结构：

### 6.1 项目简介
用 1 段话说明：
- 这是一个什么项目。
- 面向什么领域问题。
- 为什么要做。

### 6.2 当前定位
用 3-5 条简短信息说明：
- 当前项目处于规划中还是开发中。
- 当前目标版本。
- 当前最高优先级。
- 当前仓库主要存放什么内容。

### 6.3 当前状态总览
使用表格展示当前真实状态，列建议为：
- 模块
- 当前状态
- 说明
- 下一步

建议覆盖：数据源、切分、检索、评估、模型、部署、文档。

### 6.4 当前阶段与下一步
这是首页的核心区块，直接回答“现在该做什么”。

建议包含：
- 当前阶段：例如 `v1.0 Baseline 规划中`。
- 本周重点：1-3 项。
- 下一步任务：3-5 项可执行事项。
- 优先级提醒：例如先保 v1.2 检索层收益。

### 6.5 开发路线图
保留简化版路线图表格，仅包含：
- 版本
- 核心目标
- 关键产出
- 状态

详细描述移到 `docs/roadmap.md`。

### 6.6 文档导航
首页提供文档索引，而不是把全部正文塞在首页。

建议导航项：
- `docs/roadmap.md`：详细版本规划
- `docs/evaluation.md`：评估框架与指标口径
- `docs/resume_notes.md`：简历表达与面试亮点
- `docs/references.md`：参考资料与学习路径

### 6.7 成功标准
只保留精简版量化目标，按以下维度列出目标值或方向：
- 检索效果
- 回答质量
- 系统性能
- 领域覆盖

此处必须明确这些是目标，不是当前已达成结果。

### 6.8 仓库说明
说明当前仓库状态，例如：
- 当前以规划文档为主。
- 后续代码、脚本、结果文件会随版本推进逐步补齐。
- README 会持续从“规划首页”演进为“项目首页”。

## 7. docs 文档拆分设计
首次重构时，直接按主题拆出以下文件：

1. `docs/roadmap.md`
   - 承接现有 README 中的版本规划、技术决策、风险与时间管理。
2. `docs/evaluation.md`
   - 承接评估策略、数据构建原则、指标说明。
3. `docs/resume_notes.md`
   - 承接简历表达、项目亮点、面试准备。
4. `docs/references.md`
   - 承接参考资料、学习路径、外部链接。

本次设计不采用最小化二拆分方案，原因是该方案会继续混合不同主题，后续维护成本更高。按主题拆分虽然文件更多，但边界更清晰，更符合本次首页重构的目标。

## 8. 内容迁移映射
现有 README 内容按下列方式迁移：

- 项目定位、目标时间、核心原则 → `README.md` 首页保留精简版。
- 项目现状分析 → `README.md` 保留精简状态表。
- 版本规划（v1.0-v1.4） → `README.md` 保留摘要，详细版迁移到 `docs/roadmap.md`。
- 技术决策详述 → 统一迁移到 `docs/roadmap.md`；其中评估指标定义、评估数据构建原则与评估流程单独整理到 `docs/evaluation.md`。
- 架构图 → `README.md` 保留简化版架构概览，完整说明放 `docs/roadmap.md`。
- 开发任务清单 → `docs/roadmap.md`。
- 风险与应对、时间管理 → `docs/roadmap.md`。
- 成功标准 → `README.md` 保留精简版。
- 简历呈现、面试亮点 → `docs/resume_notes.md`。
- 参考资料、学习路径 → `docs/references.md`。

## 9. 内容表达规则
重构后的首页需要统一遵守以下表达规则：

1. 对当前仓库真实存在的内容，用“当前状态”口径描述。
2. 对尚未实现但近期准备推进的内容，用“进行中”或“下一步”口径描述。
3. 对更远期内容，用“计划中”口径描述。
4. 不出现把目标值、规划结果写成已完成成绩的表述。
5. 不编造快速开始命令；如果后续没有可运行入口，则以“仓库当前说明 + 文档导航”为主。
6. 首页优先保留行动导向信息，长篇背景材料一律下沉。

## 10. 预期收益
完成重构后，首页将带来以下收益：
- 能快速帮助作者判断当前阶段与下一步行动。
- 能保留项目整体技术方向，避免开发中偏航。
- 能显著降低 README 的阅读负担。
- 能为后续逐步把规划替换成真实实现结果留下清晰演进路径。

## 11. 实施边界
本次后续实施应只包含以下范围：
1. 重写 `README.md` 首页结构与文案。
2. 新建并迁移必要的 `docs/*.md` 文档。
3. 不修改项目技术方案本身。
4. 不补造代码、脚本、评估结果或运行说明。

## 12. 验收标准
如果重构后的首页满足以下条件，则认为设计达成目标：
1. 用户在 1 分钟内能看明白项目是什么、现在在哪个阶段、下一步做什么。
2. 首页长度明显下降，但仍保留项目全貌的导航能力。
3. 读者能明确区分现状、进行中事项和计划中事项。
4. 简历表达、学习资料等内容不再干扰首页的开发导向。
5. 详细内容在 `docs/` 中可被顺畅找到。
