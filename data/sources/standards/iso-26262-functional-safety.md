# ISO 26262 功能安全标准

## 概述

ISO 26262 是国际标准化组织（ISO）发布的针对道路车辆功能安全的核心标准，全称为"Road vehicles — Functional safety"。该标准源自更广泛的 IEC 61508 功能安全标准，专门针对汽车电子电气（E/E）系统的功能安全需求进行了定制化修订。ISO 26262 为汽车行业的整个安全生命周期提供了系统化的框架，涵盖从概念设计、系统开发、硬件与软件实现、生产运行到最终退役的全过程。

该标准的核心目标是确保电子电气系统在出现故障时能够进入安全状态或维持安全运行，从而避免因系统故障导致的不合理人身伤害风险。对于自动驾驶系统而言，ISO 26262 提供了基础性的安全保障框架，是实现车辆功能安全的基本要求。

## 适用范围

ISO 26262 适用于安装在量产乘用车（最大总质量不超过 3500 kg）中的电气/电子（E/E）系统。标准涵盖以下关键领域：

- **电子控制单元（ECU）**：包括发动机控制、制动控制、转向控制等关键系统
- **传感器与执行器**：雷达、摄像头、激光雷达（LiDAR）等感知系统
- **软件组件**：嵌入式软件、控制算法、决策逻辑
- **电气布线系统**：车载网络、通信总线、电源分配系统
- **自动驾驶相关系统**：ADAS（高级驾驶辅助系统）、自动驾驶决策与控制系统

需要注意的是，ISO 26262 不直接覆盖机械故障、电磁兼容性（EMC）或网络安全威胁，这些领域需要通过其他标准和规范进行管理。

## 核心要求

### 安全生命周期管理

ISO 26262 要求对产品实施完整的安全生命周期管理，主要包括以下阶段：

1. **概念阶段（Part 3）**：进行危害分析与风险评估（HARA），确定安全目标（Safety Goals），分配汽车安全完整性等级（ASIL）
2. **产品开发（Parts 4-6）**：按照 V 模型进行系统级、硬件级和软件级的开发与验证
3. **生产与运行（Part 7）**：制定安全相关的生产计划和运行维护指南
4. **支持过程（Part 8）**：配置管理、变更管理、文档管理等支撑过程
5. **ASIL 分解（Part 9）**：通过 ASIL 分解降低单一组件的安全等级要求

### 硬件架构度量指标

对于硬件设计，标准定义了两个关键的架构度量指标：

- **单点故障度量（SPFM, Single Point Fault Metric）**：衡量系统对单点故障的容忍能力，ASIL D 要求 ≥ 99%，ASIL C 要求 ≥ 97%
- **潜伏故障度量（LFM, Latent Fault Metric）**：衡量系统对潜伏故障的检测能力，ASIL D 要求 ≥ 90%，ASIL C 要求 ≥ 60%
- **随机硬件失效概率目标（PMHF）**：ASIL D 要求 < 10 FIT（Failure in Time），ASIL B 要求 < 100 FIT

### 软件开发要求

软件开发需遵循严格的过程要求，包括：

- 需求规范的双向可追溯性
- 架构设计中安全机制的实现
- 使用受控的编码规范（如 MISRA C/C++）
- 静态代码分析与动态测试相结合
- 单元测试、集成测试和系统测试的全面覆盖

## 安全等级定义（ASIL）

ISO 26262 引入了汽车安全完整性等级（ASIL, Automotive Safety Integrity Level）作为核心分级机制，从 ASIL A（最低）到 ASIL D（最高），以及 QM（质量管理，无额外安全要求）。ASIL 等级通过以下三个因素综合确定：

| 因素 | 说明 | 等级划分 |
|------|------|----------|
| 严重性（Severity） | 事故后果的严重程度 | S0（无伤害）至 S3（致命伤害） |
| 暴露概率（Exposure） | 危险场景发生的可能性 | E0（极不可能）至 E4（高频） |
| 可控性（Controllability） | 驾驶员控制危险情况的能力 | C0（可控）至 C3（不可控） |

典型的 ASIL 等级分配示例：
- **ASIL D**：自动驾驶系统的感知与决策模块、电动助力转向系统
- **ASIL C**：自适应巡航控制（ACC）、自动紧急制动（AEB）
- **ASIL B**：车道偏离预警（LDW）、自适应远光灯
- **QM**：车内娱乐系统、座椅调节系统

## 验证与确认方法

ISO 26262 要求采用多层次、系统化的验证与确认方法：

- **需求验证**：通过评审、检查和形式化方法确保需求的正确性、完整性与一致性
- **设计验证**：使用 FMEA（失效模式与影响分析）、FTA（故障树分析）等方法验证架构设计的安全性
- **硬件验证**：通过 SPFM、LFM 度量和 PMHF 计算验证硬件设计满足目标等级
- **软件验证**：单元测试（语句覆盖、分支覆盖、MC/DC 覆盖）、集成测试、回归测试
- **系统集成测试**：在硬件在环（HIL）、车辆在环（VIL）环境中进行端到端验证
- **安全确认**：通过测试、分析和现场运行数据确认安全目标已达成

对于自动驾驶系统，标准还鼓励采用仿真测试、场景测试和真实道路测试相结合的综合验证策略。

## 与其他标准的关系

ISO 26262 是自动驾驶安全标准体系的基石，与其他标准形成互补关系：

- **ISO 21448（SOTIF）**：ISO 26262 关注系统故障导致的安全问题，而 ISO 21448 关注系统在无故障情况下因功能不足或人员误用导致的安全风险。两者共同构成自动驾驶安全的完整保障框架。
- **UL 4600**：UL 4600 在 ISO 26262 的基础上，进一步扩展了对高度自动驾驶系统整体安全评估的要求，包括安全论证（Safety Case）和持续运行安全保障。
- **ISO/PAS 21448 与 ISO 26262 的协同应用**：在实际项目中，两个标准通常需要同时应用。ISO 26262 处理硬件随机失效和系统性故障，SOTIF 处理传感器性能局限、算法边界情况等预期功能不足问题。
- **ANSI/ATA 标准**：美国的 ANSI/ATA 标准体系在 ISO 26262 的基础上，增加了针对美国法规环境和运营场景的特定要求。
- **中国标准体系**：中国的自动驾驶标准（如 GB/T 标准）在制定过程中大量参考了 ISO 26262 的框架，并结合中国特有的交通环境进行了本地化调整。

ISO 26262 的最新版本（第二版，2018 年发布）已将卡车、公共汽车和摩托车纳入适用范围，并增强了对半导体器件和软件重用的指导，更好地适应了自动驾驶技术快速发展的需求。
